# 我的地图（用户中心）交互优化方案（第一性原理）

## 1. 问题复盘（基于现状）

当前地图详情页存在以下可感知问题：

1. **信息架构重复**：页面同时存在「当前地图待排」与「全局想去池」，并且全局池卡片还提供「加入当前地图」动作，导致用户先理解两层容器，再理解迁移动作，认知负担偏高。  
2. **交互路径过长**：主流程高度依赖拖拽；当点位多时，用户需要在长列表中反复上下拖动，操作成本高，且易卡顿。  
3. **意图不一致**：路线预览区地图本身可点击跳转 Google Maps，同时又提供「在 Google Maps 打开」按钮与「进入站内导航」，主次不清，行为重复。  
4. **流程状态过多**：当前页面同时承载编辑、排序、路线预览、逐段导航、打卡等任务，单页心智模型复杂。  

## 2. 第一性原理拆解

围绕“用户在我的地图页到底要完成什么”可还原成三件事：

- **收集点位**：把“以后可能去”的点保存在个人空间。
- **组织行程**：从点位中选出“这次要去”的路线并排序。
- **执行导航**：一键跳转到外部地图执行，不在站内做重型导航替代。

据此，产品应满足三条硬约束：

1. **单一数据源**：避免“全局池 + 当前地图池”双重真相。
2. **最短可达路径**：常用动作（加入路线、排序、导航）应 1~2 步完成。
3. **低性能成本交互**：减少全量拖拽依赖，优先点击式排序与局部拖拽。

## 3. 目标态（建议 IA 与交互）

### 3.1 信息架构重组

将页面改为“三栏逻辑，两层优先级”：

- **左侧：我的点位（唯一池）**  
  - 数据来源统一为用户全局点位。  
  - 不再区分“当前地图待排”和“全局想去池”。
- **中间：本次路线（当前地图）**  
  - 仅展示已加入当前地图的点位，支持轻量排序。
- **右侧：导航摘要**  
  - 只保留核心摘要（总点数/预计时长可后续接入）+ 一个主 CTA「去 Google Maps 导航」。

### 3.2 主流程精简

- 加点：`我的点位 -> 加入路线`（点击按钮）
- 排序：默认使用 `上移/下移/置顶/置底` 快捷动作；拖拽仅作为增强能力。
- 导航：始终单一主按钮 `去 Google Maps 导航`。

### 3.3 拖拽降级策略

- 仅在“本次路线”短列表内启用拖拽（例如 <= 30 项）。
- 长列表自动切换为点击排序模式（批量上移/下移 + 快速定位）。
- 对跨区移动（池 -> 路线）取消“远距离拖拽”，统一改为点击加入。

### 3.4 预览与跳转策略

- 预览地图保留为“只读反馈”，文案明确「预览仅供确认，导航请点击下方按钮」。
- 移除重复入口：
  - 若地图区域本身可点击，则取消旁路“在 Google Maps 打开”次要按钮；
  - 或保留按钮为唯一入口，同时地图不再可点击（二选一，建议后者，误触更低）。

## 4. 分阶段落地计划

### Phase 0（1 周）——无后端改造的快赢

- 文案统一：将「全局想去池」与「当前地图待排」文案明确成“我的点位/路线点位”。
- 导航入口去重：仅保留一个主导航 CTA。
- 排序快捷按钮：在路线卡片补充上移/下移，降低拖拽依赖。
- 增加“快速回到顶部/定位到路线区”悬浮操作，缓解长距离拖动。

**验收指标**：
- 地图页平均操作时长下降（例如 -20%）；
- 拖拽操作占比下降，点击式操作占比上升；
- 路线导出/导航点击率提升。

### Phase 1（1~2 周）——数据模型与交互统一

- 页面层取消双池展示，重构为“唯一点位池 + 当前路线”。
- `加入当前地图`改为显式状态切换（未加入/已加入）。
- 提供筛选器：未加入、已加入、已打卡。

**验收指标**：
- 新用户首次成功规划路线时间下降；
- 与“点位属于哪个池子”相关的客服/反馈显著减少。

### Phase 2（2 周）——性能与可扩展

- 列表虚拟化（仅渲染可视区）+ 预览请求分批策略优化。
- 大列表禁用跨区拖拽，保留轻量 reorder。
- 增加批量操作（批量加入路线、批量移出、按距离或收藏时间排序）。

**验收指标**：
- 千级点位下滚动 FPS 稳定；
- 拖拽掉帧与误操作率下降；
- 长列表任务完成率提升。

## 5. 面向当前代码的改造建议（实现导向）

1. **拆分职责组件**：将当前单文件的数据拉取、拖拽、导航、打卡逻辑拆成 `PointPoolPanel / RoutePanel / NavigationPanel`，降低耦合。  
2. **统一动作语义**：`handleAddFromPointPool` 与 `handleMoveToZone` 抽象为 `setPointMembership(pointId, inRoute)`，避免“池子驱动”思维泄漏到 UI。  
3. **拖拽逻辑收敛**：`handleDragEnd` 目前分支较多，建议仅保留“路线内部 reorder”分支，跨区改点击动作。  
4. **预览入口单一化**：地图区域与按钮保持唯一跳转入口，避免双入口竞态和认知冲突。  

## 6. 风险与回滚

- 风险：老用户对拖拽路径有习惯，突然移除会产生短期不适。
- 策略：先灰度（10% 用户）+ 开关控制新旧交互并行一版。
- 回滚：保持原 API，不改核心数据结构，主要在前端展示层切换，回滚成本低。

## 7. 推荐优先级（结论）

按投入产出比，建议优先顺序：

1. **先做入口去重 + 点击排序（立刻减负）**
2. **再做双池合一（根因治理）**
3. **最后做性能深化（虚拟化/批量操作）**

## 附：与当前页面实现的对应关系

- 当前拖拽逻辑覆盖排序、跨区迁移、池子入路线等多个分支。
- 当前预览区同时存在可点击地图与外链按钮，属于功能重复。
